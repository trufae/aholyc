# aholyc internals

## Design ideas borrowed from TempleOS

TempleOS's compiler is ~20k lines of HolyC that lexes, parses and emits
x86-64 in basically one pass, with no external anything. aholyc keeps the
values that made that possible:

* **Zero dependencies.** Building aholyc needs `cc` and `make`. The runtime
  and prelude are embedded into the binary at build time (`tools/file2c`),
  so the installed compiler is one self-contained file.
* **One shared header.** `src/aholyc.h` holds every structure: tokens,
  types, AST, symbols, the backend vtable. No abstraction layers, no
  visitor patterns — plain switch statements over node kinds.
* **Lower early, keep backends dumb.** TempleOS pushed complexity into
  the front of the compiler; aholyc's parser desugars everything HolyC-
  specific (default args, implicit prints, sub_switch, `++`, chained
  comparisons) so each backend is a straightforward tree-walker.
* **Let the big hammer swing.** TempleOS trusted its own passes; aholyc
  emits naive alloca-style IR and trusts LLVM/cc `-Os`. Simple emitter,
  good code anyway.
* **Small binaries.** `-Os`, dead-code elimination, and one tiny runtime.

## Pipeline

```
file.HC ──lex──▶ tokens ──preprocess──▶ tokens ──parse──▶ Program ──emit──▶ .ll/.c/.js ──build──▶ exe
              (#include,#define,#ifdef,#exe)    (AST+lowering)        (external toolchain)
```

* `src/lex.c` — tokenizer + token-level preprocessor. Multi-char
  constants (`'ABC'`), the `` ` `` operator, `0x`/`0b` literals.
  `#include` splices token streams; macros are token lists with a
  self-reference guard. `#exe{...}` bodies are cut out of the stream,
  run by `src/exe.c`, and their `StreamPrint` output is re-tokenized
  and spliced back in.
* `src/exe.c` — compile-time execution: the block becomes a normal
  program (same parser, C backend), built as a shared library and
  dlopened into the compiler process. `parse()` resets its state at
  entry so this nested compile can precede the outer one. See
  `doc/exe.md`.
* `src/parse.c` — recursive descent following TempleOS's operator
  precedence. Produces the core AST and does all lowering:
  * top-level statements + global initializers → the startup function
    `__hc_start`, executed in source order (rt's `main` calls it);
  * `switch` → compare/goto dispatch (sub_switch porches become a
    per-switch selector temp and a second dispatch after the porch);
  * loops get explicit end labels so `break` is just `goto`;
  * default arguments are spliced into each call site;
  * `x op= y`, `x++` → assignments via address temps (single evaluation);
  * `a<b<c` → `(a<(t=b)) && (t<c)`;
  * assignment of narrow types yields the pre-truncation value via temps
    (TempleOS semantics).
* `src/back_*.c` — see `doc/backends.md`.
* `src/main.c` — the driver: options, backend registry, toolchain
  invocation.
* `runtime/rt.c`, `runtime/rt.js` — the runtime; `runtime/prelude.hc`
  declares it to the compiler; `runtime/exe.hc` is the extra prelude
  for `#exe` blocks. All four are embedded via `src/embed.c`,
  generated by `tools/file2c` in the Makefile.

## Calling convention

All HolyC scalars travel as 64-bit values (`i64`/`double`). Variadic
functions follow the TempleOS model rather than C varargs: the compiler
packs the extra arguments into an on-stack array of 8-byte slots and
passes `(named..., I64 argc, I64 *argv)`. F64 varargs are bit-copied into
their slot; `%f` in `Print` reads the slot as a double. This sidesteps C
vararg ABI headaches and gives HolyC's `argc`/`argv` for free.

## Exceptions

The native runtime keeps `Fs` and a fixed stack of `jmp_buf` frames in
thread-local storage. `Fs` is lazily initialized to a private `CTask` for
each host thread. `try` pushes a frame and `_setjmp`s; `throw(ch)` records
`ch` in `Fs->except_ch`, pops the top frame and longjmps into its catch
block. The catch code decides: setting `Fs->catch_except` (what
`PutExcept` does) consumes the exception; otherwise the generated code
rethrows to the next outer frame — exactly TempleOS's handler-search
semantics. The JS backend maps this onto native JS exceptions instead.

## Testing

`make test` → `tests/run.sh` compiles every `examples/*.HC` with every
backend available on the machine and diffs the output against
`tests/expected/*.out`. The C backend is the reference; the LLVM and JS
backends must match it byte for byte.
